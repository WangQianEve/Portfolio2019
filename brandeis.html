Summary
Problem
How might we help people protect their mobile privacy easily in both personal and organizational settings?
Current mobile systems let users control each app’s access to sensors/data at a certain level. However, this is not enough. Based on our research, many applications request more data than they need, or even leak user data to other companies. This can jeopardize the user’s properties, or even put lives under threat for those working in particular fields (e.g., military).
Solution
A privacy-enhanced Android system that helps you control and understand your privacy settings.
We designed 5 different touchpoints to configure privacy settings. Privacy manager is the most important one, it is a centralized configuration app. The 5 touchpoints together manages 4 layers of settings.
See details below.

Solution
Touch Point 1 - Privacy Manager

Privacy Manager is where users can manage their privacy settings.

Homepage

From the homepage, users can access global settings, app settings, and privacy modes. Besides that, there’s a privacy overview section showing how users’ data is currently being accessed. There's also a recommended setting section to help users quickly configure their settings.

Global Setting
“What-Why-Where” Setting
Users can manage access globally in 3 levels: “what data is being requested,” “why it’s requested,” and “where is the data used.” This order is determined through a user research method called "card sorting". The settings are presented in a progressive disclosure fashion.
Privacy Mode
Organizational Need

Organizations often need to prevent data leaks from its employees or visitors. For instance, DARPA’s military people may need to hide their location from all apps when they are out on a mission. The privacy mode allows users to apply different set of permissions in different contexts.

App Settings

Right after users installed an app, the App Settings page will be prompted. By default, all permissions are set based on your usual settings. If no previous settings were found, they're set based on settings of the majority of PE-Android users.

Uncommon permission requirements are put upfront to attract users' attention.

Run-time Message

While users are using an app, a pop-up will appear if the requested permission is set to “Ask”. Users can view the detailed explanation of why the app needs the data by clicking “Show More”. Checking “Never Ask Again” will correspondingly change the App privacy setting.

Notification Center

Sometimes, users need a temporary exception to the privacy setting to use certain functionality. They can view the most recently blocked request in the notification center, then choose to allow just once or open the privacy manager to change the setting.

Success Metrics
We did thorough iterations through out the 5 months to make sure this complex system is easy to use and understand. We have 5 ~ 15 iterations for each design decision we made. Following a double diamond practice, we often generate multiple designs, and then narrow down through user testing. On average, we conducted at least 3 user testings per week. This give us confidence to have a conversation with Google for future cooperation.
Ease of Use
Multiple touchpoints and multiple layers of settings made system complicated. We need to simplify the logic and express it clearly through the design.
Support Various User Types
People have different needs for privacy control. Some of them want more fine-grained control, while others want less. Employees' need may change based on contexts. We need to design for all of them.
Align with Material Design
This project is based on Android. Considering both user habit and development requirements, we need to follow Material Design, and know when to innovate for a better user experience.

Research Methods

Since the project already started when I joined. We did a little generative research, then mainly focused on evaluative research.

Card Sorting
Decide the information hierarchy: "what-why-where"

We used card sorting to figure out the right information hierarchy. We printed out permission cards with "what data this is for", "why it was requested", and "who'll be using it" on each one. Then we asked users to organize them. The result shows 80% of users sorted them in a “what-why-where” sequence.

card sorting
A/B Testing + Think-Alouds
Keep refining the design until it's easy to use for different people.

Designing for such a complicated system and various user types means testing again and again. The most often used method is the simplest A/B testing combined with think-alouds. For each design decision, we usually develop multiple designs to test. Asking users to perform a task and thinking aloud helped us better understand the mindset of users, thus informed us how to proceed.

an example of a/b testing
Dev Check / Developer Walkthrough
Data-driven design: results in a progressive disclosure fashion.

Figuring out exactly what and how much data will be shown on each page helped us refine the design. For instance, we applied a progressive disclosure fashion for global setting. This is because after checking the log data, we realized that there could too long a list. We also walked through the design with developers to make sure all our designs are feasible under the current Android framework.

Android's developer documentation about permissions
Designer Walkthrough / Heuristic Evaluation
The design should be consistent, align with Material Design, and covers edge cases.

Besides user testing, we also conduct designer walkthroughs once in a while to make sure: 1. our design is consistent. 2. it aligns with Material Design. 3. it covered as much edge cases as possible. These are all key metrics for the success of this project.

04
Reflection
Things we did correctly 😉
  • The key to designing a complex system is to simplify it through understanding the data and logic. We spent weeks discussing the control logic, conducting user research to understand users’ mental models before settling down on a design.
  • We had a good balance between following a design guideline (Material Design) and innovating based on the need.
Things we can improve 🤔
  • We couldn't test the privacy profile with organizational users due to lack of access. If possible, I hope to do more testing with them.
  • If given more time, I hope to further explore how we can recommend configurations, thus making it even easier to maintain a good setting.
