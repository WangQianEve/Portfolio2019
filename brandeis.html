Problem
How might we help users protect their mobile privacy easily in various contexts and satisfy both individual and organizational needs?
Current mobile systems let users control each app’s access to sensors/data at a certain level. However, this is not enough. Based on our research, many applications request more data than they need, or even leak user data to other companies. This can jeopardize the user’s properties, or even put lives under threat for those working in particular fields (e.g., military).
Solution
A privacy-enhanced Android system that helps you control and understand your privacy settings.
How Does It Work?
Touch Point 1 - Privacy Manager

Privacy Manager is where users can manage their privacy settings.

Homepage

From the homepage, users can access their global settings, app settings, and privacy profiles. Besides that, there’s a privacy overview section that provides a brief view of how users’ data is currently being accessed.

Recommended Setting
The system also generates recommended settings so that users can improve their privacy settings easily.

Global Setting

“What-Why-Where” setting
Our privacy-enhanced Android system requires app developers to specify the purpose of every data request, and can trace back which specific libraries used the data. So we used a “what-why-where” information hierarchy: “what data is being requested,” “why it’s requested,” and “where is the data used.” They are shown in a progressive disclosure fashion in both global settings and app settings.

Privacy Mode

Organizations sometimes need to prevent data leaks from its employees or visitors. For instance, DARPA’s employees sometimes need to hide their location from all apps when they are out on a mission. So we designed the privacy mode, which contains a set of permission rules that users cannot change.

Touch Point 2 - Quick Setting
Quick Setting

PE-Android supports more privacy-related options in the quick setting view. This can help those who are in a sensitive situation and want to temporarily block specific types of data access.

In the example on the right, certain permissions are disabled by the current privacy mode.

Touch Point 3 - App Installation
App Settings

Right after users installed an app, the App Settings page will be prompted. By default, all permissions are set based on your usual settings. If no previous settings were found, they're set based on settings of the majority of PE-Android users.

Uncommon permission requirements are put upfront to attract users' attention.

In the example on the right, certain permissions cannot be granted because they're disabled by the current privacy mode.

Touch Point 4 - App Running
Run-time Message

While users are using an app, a pop-up will appear if the requested permission is set to “Ask”. Users can view the detailed explanation of why the app needs the data by clicking “Show More”. Checking “Never Ask Again” will correspondingly change the App privacy setting.

Touch Point 5 - Need Exception
Notification Center

Sometimes, users need a temporary exception to the privacy setting to use certain functionality. They can view the most recently blocked request in the notification center, then choose to allow just once or open the privacy manager to change the setting.

Main Challenges
Complicated System
Multiple touchpoints complicated the system. We need to stand in users’ shoes to simplify the logic and use design to express it clearly. Our final solution is:
1. Privacy mode has the highest control, quick settings come next, then global settings and app settings.
2. Between global settings and app settings, the most recent change is effective.
Various User Types
As a mobile OS, we have various target user types. Some of them want more fine-grained control, while others may not even want to configure anything. In the end, we used progressive disclosure to serve various types of users.
Apply Existing Design Style
Our work is based on the Android system. Considering both user habit and development constraints, we need to follow a design style similar to that used in Android. However, existing styles are not always suitable for our features. When and how to innovate while staying close to the design style is a challenge.
Research Methods
Card Sorting
We used card sorting to figure out the right information hierarchy. The result is our “what-why-where” structure.
A/B Testing
With a complicated system and various user types, we often come up with multiple potential designs. A/B testing helped us identify the strengths and weaknesses of each, and inspired new ideas.
Cognitive Walkthrough
The more complex the system is, the more user testing and walkthrough it needs. We’ve been continuously doing cognitive walkthroughs with users from different backgrounds during this project span to avoid missing edge cases.
Reflection
What did I learn?
  • The key to designing a complex system is to simplify it through understanding the data and logic. We spent weeks discussing the control logic, conducting user research to understand users’ mental models before settling down on a design.
  • Applying a design style like Material Design is not only about imitation; it’s also about innovation. During prototyping, we often need to refer to the Material Design website and examples in the latest Android system to seek inspiration, while also reading through the Android developer documentation to figure out feasibility. It is a challenge to create a design based on an existing style while still fitting into the concept.
How can this project proceed?

This project is still ongoing, we are in the process of talking to more sponsors and partners. In the future, we hope to keep refining the product until it can satisfy more users and more institutional needs. One direction I’m particularly interested in is to provide smarter recommendations for privacy settings so that users don’t have to dive into details of the underlying system.

I hope one day, everyone can protect and manage their privacy easily and wisely.
